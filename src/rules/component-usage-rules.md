# Component Usage Rules

## Button

| Variant | استفاده |
|---|---|
| Filled | اکشن اصلی مثل تأیید، ذخیره، ادامه |
| Outlined | اکشن ثانویه مثل بازگشت یا لغو |
| Text | اکشن‌های کم‌اهمیت مثل بیشتر، رد کردن |
| Elevated | زمانی که دکمه روی سطح نیاز به elevation دارد |
| Tonal | اکشن متوسط، کمتر از Filled اما مهم‌تر از Text |

## ممنوعیت‌ها

- ساختن دکمه با div ممنوع است.
- استفاده از رنگ مستقیم مثل `#65558f` داخل کامپوننت ممنوع است.
- حذف focus state ممنوع است.
- حذف disabled state ممنوع است.


## TextField

| Variant | استفاده |
|---|---|
| Filled | فرم‌های معمولی، فیلدهای پرکاربرد، فرم‌های داخل سطح‌ها |
| Outlined | فرم‌هایی که نیاز به مرزبندی واضح‌تر دارند |
| Error | وقتی مقدار نامعتبر است؛ باید `errorText` داشته باشد |
| Disabled | وقتی فیلد قابل ویرایش نیست |

## قوانین TextField

- ارتفاع پایه باید 56px باشد.
- Label باید از Body Small استفاده کند.
- متن ورودی باید از Body Large استفاده کند.
- Supporting text باید فقط از طریق prop استاندارد بیاید.
- ساخت input خام در صفحه ممنوع است؛ باید از TextField استفاده شود.
- برای موبایل، TextField معمولاً باید `fullWidth` باشد.


## Card

| Variant | استفاده |
|---|---|
| Filled | کارت‌های معمولی و محتوای عمومی |
| Outlined | وقتی جداسازی محتوایی باید واضح‌تر باشد |
| Elevated | وقتی کارت باید تاکید یا اولویت بصری بیشتری داشته باشد |

## قوانین Card

- برای container کارت از `Card` استفاده شود، نه div خام.
- variantهای مجاز: `filled`, `outlined`, `elevated`.
- stateهای تعاملی باید از `interactive` یا `state` بیایند.
- stateهای قابل پشتیبانی مطابق فیگما: `enabled`, `hovered`, `focused`, `pressed`, `dragged`.
- برای کارت‌های کلیک‌پذیر باید focus-visible وجود داشته باشد.
- عنوان کارت باید از `title` بیاید، نه h3 دستی داخل children.


## Switch

| حالت | استفاده |
|---|---|
| Checked | گزینه فعال است |
| Unchecked | گزینه غیرفعال است |
| Disabled | گزینه قابل تغییر نیست |

## قوانین Switch

- Switch فقط برای انتخاب دودویی on/off استفاده شود.
- برای چند انتخاب مستقل، Checkbox مناسب‌تر است.
- برای انتخاب یکی از چند گزینه، Radio مناسب‌تر است.
- همیشه باید label قابل فهم داشته باشد.
- input native checkbox باید حفظ شود تا accessibility درست بماند.
- در RTL، حرکت handle باید با چیدمان راست‌به‌چپ سازگار باشد.


## Checkbox

| حالت | استفاده |
|---|---|
| Unchecked | گزینه هنوز انتخاب نشده است |
| Checked | گزینه انتخاب شده است |
| Indeterminate | بخشی از زیرگزینه‌ها انتخاب شده‌اند |
| Error | انتخاب اجباری یا مقدار نامعتبر است |
| Disabled | گزینه قابل تغییر نیست |

## قوانین Checkbox

- Checkbox برای انتخاب چند گزینه مستقل استفاده شود.
- برای انتخاب on/off تکی، Switch مناسب‌تر است.
- برای انتخاب فقط یک گزینه از چند گزینه، Radio مناسب‌تر است.
- native checkbox input نباید حذف شود.
- حالت indeterminate باید فقط وقتی استفاده شود که انتخاب بخشی از زیرمجموعه‌ها معنی دارد.
- اگر checkbox اجباری است و انتخاب نشده، باید errorText داشته باشد.


## Radio Button

| حالت | استفاده |
|---|---|
| Unselected | گزینه انتخاب نشده است |
| Selected | گزینه انتخاب شده است |
| Error | انتخاب الزامی یا نامعتبر است |
| Disabled | گزینه قابل تغییر نیست |

## قوانین Radio

- Radio برای انتخاب فقط یک گزینه از چند گزینه استفاده شود.
- Radioها باید داخل `RadioGroup` استفاده شوند.
- برای انتخاب چند گزینه مستقل، Checkbox مناسب‌تر است.
- برای on/off تکی، Switch مناسب‌تر است.
- native radio input نباید حذف شود.
- هر Radio باید value مشخص داشته باشد.
- هر RadioGroup باید name مشترک داشته باشد؛ اگر name داده نشود، کامپوننت خودش تولید می‌کند.


## Chip

| Variant | استفاده |
|---|---|
| Assist | اکشن کمکی یا پیشنهاد هوشمند |
| Filter | انتخاب و حذف فیلترها |
| Input | نمایش ورودی کاربر مثل تگ یا انتخاب‌شده‌ها |
| Suggestion | پیشنهاد سریع برای جستجو یا انتخاب |

## قوانین Chip

- Chip جایگزین Button اصلی نیست.
- برای اکشن اصلی صفحه از Button استفاده شود.
- Filter Chip باید selected state داشته باشد.
- Input Chip اگر قابل حذف است باید onRemove داشته باشد.
- برای انتخاب‌های پیچیده از Checkbox یا Radio استفاده شود.
- Chipها باید با keyboard قابل فوکوس و فعال‌سازی باشند.


## Tabs

| Variant | استفاده |
|---|---|
| Primary | دسته‌بندی سطح بالا در یک صفحه |
| Secondary | دسته‌بندی سطح دوم یا داخل بخش‌ها |

## قوانین Tabs

- Tabs برای جابه‌جایی بین بخش‌های مرتبط استفاده شود.
- Tabs برای navigation اصلی کل سایت استفاده نشود؛ برای آن NavigationBar یا Drawer مناسب‌تر است.
- هر Tab باید value یکتا داشته باشد.
- هر TabPanel باید value متناظر با Tab داشته باشد.
- active tab باید indicator واضح داشته باشد.
- disabled tab نباید قابل کلیک باشد.
- در صفحات فارسی، Tabs باید RTL-friendly باشد.


## Top App Bar

| Variant | استفاده |
|---|---|
| Small | اکثر صفحات معمولی |
| Center | صفحات ساده با عنوان وسط‌چین |
| Medium | صفحات با تاکید بیشتر روی عنوان |
| Large | صفحات اصلی یا سطح بالاتر |

## قوانین Top App Bar

- Top App Bar برای هدر صفحه یا اپلیکیشن استفاده شود.
- برای navigation اصلی طولانی، Drawer یا Navigation Bar مناسب‌تر است.
- navigation icon باید aria-label داشته باشد.
- actionها باید label قابل فهم داشته باشند.
- برای صفحات فارسی، title باید `text-align: start` داشته باشد مگر در variant center.
- اگر صفحه اسکرول می‌شود، می‌توان elevated/scrolled state را فعال کرد.


## Navigation Bar

| بخش | استفاده |
|---|---|
| Active item | مقصد فعلی کاربر |
| Inactive item | مقصدهای قابل انتخاب دیگر |
| Badge | تعداد پیام، اعلان یا مورد جدید |
| Disabled | مقصدی که فعلاً در دسترس نیست |

## قوانین Navigation Bar

- برای navigation اصلی در موبایل استفاده شود.
- تعداد آیتم‌ها بهتر است بین 3 تا 5 باشد.
- هر آیتم باید icon و label داشته باشد.
- active item باید indicator/container واضح داشته باشد.
- badge فقط برای اطلاعات کوتاه مثل تعداد پیام یا اعلان استفاده شود.
- برای desktop، Navigation Drawer یا Navigation Rail مناسب‌تر است.
- در RTL، ترتیب آیتم‌ها باید مطابق ترتیب طبیعی فارسی باشد.


## Navigation Drawer

| Variant | استفاده |
|---|---|
| Standard | ناوبری دائمی در دسکتاپ و تبلت |
| Modal | ناوبری موقت روی صفحه، مناسب موبایل یا حالت responsive |

## قوانین Navigation Drawer

- برای navigation اصلی در دسکتاپ و تبلت استفاده شود.
- برای موبایل، Navigation Bar یا Modal Drawer مناسب‌تر است.
- هر item باید label واضح داشته باشد.
- active item باید container واضح داشته باشد.
- badge فقط برای تعداد پیام، سفارش، اعلان یا مورد جدید استفاده شود.
- Drawer نباید برای اکشن‌های موقت استفاده شود؛ برای آن Menu یا Dialog مناسب‌تر است.
- در RTL، Drawer معمولاً از سمت راست صفحه باز می‌شود.


## List / ListItem

| Density | استفاده |
|---|---|
| One-line | آیتم‌های ساده مثل منو یا تنظیمات کوتاه |
| Two-line | آیتم با توضیح کوتاه |
| Three-line | آیتم‌های محتوایی مثل پیام، اعلان یا نتیجه جستجو |

## قوانین List

- برای مجموعه عمودی آیتم‌ها از `List` استفاده شود.
- برای هر ردیف از `ListItem` استفاده شود.
- اگر آیتم قابل کلیک است، باید `interactive` یا `onClick` داشته باشد.
- اگر آیتم انتخاب‌شده است، `selected` استفاده شود.
- متن اصلی آیتم باید در `headline` باشد.
- توضیح آیتم باید در `supportingText` باشد.
- برای تصویر، آیکن یا آواتار از `leading` استفاده شود.
- برای زمان، badge، switch یا اکشن کوچک از `trailing` استفاده شود.
- برای جداکننده‌ها از `dividers` استفاده شود، نه border دستی در صفحه.


## Dialog

| نوع استفاده | کاربرد |
|---|---|
| Alert Dialog | هشدار، تأیید حذف، تصمیم مهم |
| Basic Dialog | نمایش اطلاعات کوتاه |
| Form Dialog | فرم کوچک مثل ویرایش نام یا افزودن یادداشت |

## قوانین Dialog

- Dialog فقط برای تصمیم یا اطلاعات مهم استفاده شود.
- برای پیام کوتاه غیرمسدودکننده، Snackbar مناسب‌تر است.
- عنوان باید کوتاه و واضح باشد.
- متن توضیحی باید پیامد تصمیم را توضیح دهد.
- اکشن اصلی بهتر است در انتهای لیست actionها قرار بگیرد.
- اکشن مخرب باید متن واضح داشته باشد، مثل «حذف».
- Escape باید بتواند Dialog را ببندد، مگر در جریان‌های بسیار حساس.
- برای فرم‌های بزرگ از صفحه جداگانه استفاده شود، نه Dialog.


## Snackbar

| استفاده | توضیح |
|---|---|
| Feedback کوتاه | ذخیره شد، حذف شد، ارسال شد |
| Undo | برگشت دادن یک عملیات کوتاه‌مدت |
| Error کوتاه | خطای غیرمسدودکننده |
| Warning کوتاه | هشدار بدون نیاز به Dialog |

## قوانین Snackbar

- Snackbar برای پیام کوتاه و غیرمسدودکننده استفاده شود.
- برای تصمیم مهم یا تأیید حذف قبل از عمل، Dialog مناسب‌تر است.
- متن Snackbar باید کوتاه باشد.
- فقط یک action کوتاه مثل «بازگشت» یا «مشاهده» داشته باشد.
- Snackbar نباید جایگزین validation داخل فرم شود.
- برای خطای مهم که نیازمند تصمیم کاربر است، Dialog استفاده شود.
- duration باید به اندازه کافی باشد تا کاربر پیام را بخواند.


## Tooltip

| نوع | کاربرد |
|---|---|
| Plain Tooltip | توضیح کوتاه برای آیکن یا کنترل فشرده |
| Rich Tooltip | توضیح طولانی‌تر با عنوان و اکشن اختیاری |

## قوانین Tooltip

- Tooltip برای متن کوتاه و کمکی استفاده شود.
- Tooltip نباید اطلاعات حیاتی و ضروری فرم را پنهان کند.
- برای خطا یا validation از supporting text یا error text استفاده شود.
- برای پیام عملیاتی بعد از action، Snackbar مناسب‌تر است.
- برای توضیح آیکن‌های بدون متن، Tooltip توصیه می‌شود.
- trigger باید با keyboard هم قابل focus باشد.
- محتوای Tooltip باید کوتاه، واضح و غیرتکراری باشد.


## Menu

| استفاده | توضیح |
|---|---|
| Overflow actions | اکشن‌های بیشتر در TopAppBar یا Card |
| Contextual actions | اکشن مرتبط با یک آیتم خاص |
| Dropdown choices | انتخاب سریع از چند گزینه کوتاه |

## قوانین Menu

- Menu برای اکشن‌های کوتاه و متنی استفاده شود.
- برای فرم یا محتوای طولانی از Dialog یا صفحه جداگانه استفاده شود.
- آیتم‌های مخرب باید destructive باشند.
- آیتم disabled نباید قابل انتخاب باشد.
- برای جدا کردن گروه اکشن‌ها از divider استفاده شود.
- trigger باید aria-haspopup و aria-expanded داشته باشد.
- برای navigation اصلی از NavigationBar یا Drawer استفاده شود، نه Menu.


## IconButton

| Variant | استفاده |
|---|---|
| Standard | اکشن آیکنی کم‌تأکید |
| Filled | اکشن آیکنی با تأکید بالا |
| Tonal | اکشن آیکنی با تأکید متوسط |
| Outlined | اکشن آیکنی با مرز مشخص |

## قوانین IconButton

- برای هر دکمه فقط-آیکن باید از IconButton استفاده شود.
- هر IconButton باید label قابل فهم داشته باشد.
- برای توضیح بیشتر، IconButton می‌تواند داخل Tooltip قرار بگیرد.
- برای اکشن‌های فشرده در TopAppBar، Card و List از IconButton استفاده شود.
- اگر وضعیت انتخاب‌شده دارد، selected و selectedIcon استفاده شود.
- از button خام برای آیکن‌های قابل کلیک استفاده نشود.


## FAB

| Size | استفاده |
|---|---|
| Small | اکشن اصلی کوچک یا فضاهای محدود |
| Medium | حالت پیش‌فرض برای اکشن اصلی |
| Large | صفحات مهم یا اکشن بسیار برجسته |
| Extended | وقتی آیکن به‌تنهایی کافی نیست و label لازم است |

## قوانین FAB

- FAB فقط برای اکشن اصلی صفحه استفاده شود.
- در هر صفحه معمولاً فقط یک FAB داشته باشیم.
- برای اکشن‌های عادی از Button یا IconButton استفاده شود.
- FAB نباید جایگزین Navigation یا Menu شود.
- اگر label وجود دارد، Extended FAB استفاده شود.
- در موبایل، FAB باید با NavigationBar تداخل نداشته باشد.


## Badge

| Variant | استفاده |
|---|---|
| Small | نقطه وضعیت یا وجود اعلان بدون عدد |
| Large | شمارنده عددی پیام، اعلان یا آیتم جدید |

## قوانین Badge

- Badge برای شمارنده یا وضعیت کوچک استفاده شود.
- Badge نباید برای متن طولانی استفاده شود.
- برای مقدارهای بزرگ، max مثل 99 استفاده شود.
- Badge باید کنار کامپوننت anchor مثل IconButton یا Navigation item قرار بگیرد.
- برای پیام‌های مهم‌تر، Snackbar یا Dialog مناسب‌تر است.
- Small Badge برای وضعیت بدون عدد مناسب است.


## Divider

| Variant | استفاده |
|---|---|
| Full | جداکننده کامل بین بخش‌ها |
| Inset | جداکننده داخل List با فاصله از شروع محتوا |
| Middle | جداکننده با فاصله دوطرفه |
| Vertical | جداکننده بین اکشن‌ها یا ستون‌ها |

## قوانین Divider

- برای جدا کردن گروه‌های مرتبط از Divider استفاده شود.
- Divider نباید بیش از حد استفاده شود؛ spacing هم گاهی کافی است.
- در List بهتر است از inset divider استفاده شود.
- برای جدا کردن اکشن‌های کنار هم از vertical divider استفاده شود.
- اگر جداکننده فقط تزئینی است، decorative=true بماند.
- اگر جداکننده معنایی است، decorative=false استفاده شود.


## Slider

| حالت | استفاده |
|---|---|
| Default | انتخاب مقدار عددی ساده |
| Secondary | کنترل عددی با تأکید کمتر |
| Error | وقتی مقدار انتخابی نامعتبر است |
| Disabled | وقتی کاربر اجازه تغییر مقدار ندارد |

## قوانین Slider

- Slider برای انتخاب مقدار عددی از یک بازه استفاده شود.
- برای وارد کردن مقدار دقیق، TextField مناسب‌تر است.
- اگر مقدار دقیق مهم است، showValue یا valueFormatter فعال باشد.
- برای فیلتر قیمت، واحد پول باید در formatter نمایش داده شود.
- native range input باید حفظ شود.


## SearchBar

- برای جستجوی اصلی یک صفحه یا بخش استفاده شود.
- برای input ساده داخل فرم از TextField استفاده شود.
- اگر جستجو server-side است، loading state باید فعال شود.


## SegmentedButton

| Mode | استفاده |
|---|---|
| Single select | انتخاب یک حالت نمایش یا فیلتر |
| Multi select | انتخاب چند فیلتر همزمان |
| Compact | فضاهای محدود و toolbarها |

## قوانین SegmentedButton

- برای انتخاب بین گزینه‌های مرتبط استفاده شود.
- برای اکشن اصلی از Button استفاده شود.
- برای فیلترهای تک‌گزینه‌ای، single select مناسب است.
- برای فیلترهای چندگزینه‌ای، multi select مناسب است.
- گزینه‌ها باید کوتاه و قابل فهم باشند.


## Avatar

| Mode | استفاده |
|---|---|
| Image | تصویر واقعی کاربر |
| Initials | وقتی تصویر وجود ندارد |
| Icon | حساب عمومی، سیستم یا پشتیبانی |
| Status | نمایش وضعیت آنلاین/مشغول/آفلاین |

## قوانین Avatar

- برای کاربران، پروفایل، پیام‌ها و تیم از Avatar استفاده شود.
- اگر تصویر وجود ندارد، initials fallback مناسب است.
- برای موجودیت غیرانسانی مثل سیستم یا پشتیبانی، icon fallback مناسب‌تر است.
- status فقط وقتی استفاده شود که وضعیت حضور واقعاً معنی دارد.
- برای شمارنده‌ها از Badge استفاده شود، نه status.
